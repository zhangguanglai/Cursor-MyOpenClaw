### 单人阶段 AI 原生研发流程（需求 → 规划 → 编码 → 测试 → 上线）

> 对应待办 `workflow-spec`。本流程默认你是唯一开发者，但「人 + Agent」协作方式尽量可扩展到小团队。

---

### 1. 角色与基础环境

- **你（人类）**：
  - 产品 Owner：定义需求与验收标准。
  - 架构师：拍板关键技术决策。
  - 主开发：应用/调整 Agent 输出的代码改动。
- **系统组件**：
  - OpenClaw Core（含 LLMRouter + 各 Agent）。
  - 代码仓库（Git，本地 + 远端）。
  - Cursor / IDE。
  - CI（可从简单测试脚本开始）。

---

### 2. 整体流程总览

用一条主线描述整个闭环：

1. **需求收集**：你写下需求/Issue（自然语言）。
2. **规划（PlanningAgent）**：生成并确认实现计划与子任务。
3. **编码（CodingAgent）**：按子任务逐步生成补丁，人工审核与应用。
4. **测试与评审（TestAgent）**：生成测试建议与验收清单，执行并记录结果。
5. **上线与回顾**：合并到主干/部署，沉淀到知识库。

下面对每一步给出更详细的「操作步骤 + 输入输出规范」。

---

### 3. 需求 → 规划（Plan-first）

#### 3.1 人类操作

1. 在 `docs/cases/` 或 Issue 系统中创建一个条目，包含：
   - 功能背景与动机。
   - 粗略功能描述。
   - 业务与技术约束。
   - 初步验收标准（可以是 checklist）。
2. 在 OpenClaw Studio（或 CLI）中触发「生成实现计划」：
   - 选择/粘贴该需求描述。
   - 可选：勾选相关模块/文件，作为上下文提示。

#### 3.2 PlanningAgent 行为

- 调用：
  - 代码结构工具：了解当前项目模块。
  - 代码搜索工具：查找相关接口/模块。
- 通过 LLMRouter（如模型 `planning`）生成：
  - 详细的 Markdown 计划：
    - 涉及模块/文件。
    - 接口与数据结构变更。
    - 任务拆解与优先级。
    - 风险 & 测试建议。
  - 结构化子任务列表（写入 `plan.json` 或类似结构）。

#### 3.3 输出沉淀

- 在仓库中为该需求创建一个目录，例如：
  - `cases/feature-xxx/plan.md`
  - `cases/feature-xxx/plan.json`
- 你快速 Review：
  - 如有不合理之处，手动编辑 `plan.md` 或重新触发 PlanningAgent 进行增量调整。

---

### 4. 规划 → 编码（CodingAgent 驱动的小步改动）

#### 4.1 子任务驱动

- 对于 `plan.json` 中的每一个子任务：
  - 在 OpenClaw Studio 中选择该任务。
  - 系统自动：
    - 读取 `related_files`。
    - 从本地文件系统加载文件内容。

#### 4.2 CodingAgent 行为

- 通过 LLMRouter（如模型 `coding`）：
  - 阅读任务描述与相关文件。
  - 生成一组「候选补丁」：
    - 每个补丁包含目标文件、diff、简要说明、潜在风险与测试建议。
- 可选：对于复杂任务，可以让 CodingAgent 先给出「实现思路」再产出 Patch。

#### 4.3 人类应用与调整

- 你在 Cursor 中：
  - 审阅每个 Patch 的说明与 diff。
  - 选择性地应用/修改 Patch。
  - 如对某个补丁不满意，可在同一上下文下让 CodingAgent 重新生成或做小改动。
- 原则：
  - **保持改动的可理解性**：一次 PR/提交尽量解决一小块任务。
  - 对「含义不明的重构」保持谨慎，把更多控制权留在你手里。

#### 4.4 提交策略

- 完成一个或若干紧密相关的子任务后：
  - 本地运行基础测试。
  - 用清晰的提交信息（包含需求 ID / Case 名）提交到分支。

---

### 5. 编码 → 测试与评审（TestAgent 参与）

#### 5.1 触发时机

- 在以下时机触发 TestAgent：
  - 准备创建 PR / 合并到主干前。
  - 较大改动完成后（即使还是个人分支）。

#### 5.2 TestAgent 行为

- 输入：
  - 一组改动文件（或整次提交的 diff）。
  - 相关需求与计划文档引用。
- 通过 LLMRouter（如模型 `summary`）输出：
  - 潜在问题列表（含严重级别）。
  - 建议新增/补充的测试用例（含测试类型与步骤）。
  - 手工验收清单。

#### 5.3 人类执行与记录

- 你根据 TestAgent 输出：
  - 选择一部分测试建议实现为实际测试代码。
  - 执行推荐的自动化测试与部分手工验收。
  - 在 `cases/feature-xxx/verification.md` 中记录：
    - 已执行测试列表与结果。
    - 遇到的问题与修复记录。

---

### 6. 测试 → 上线与回顾

#### 6.1 上线步骤（个人阶段的简单版）

1. 所有关键测试通过（至少单元测试 + 手工主路径验证）。
2. 合并到主干分支（如 `main` 或 `master`）。
3. 触发部署（脚本或 CI Pipeline）。

#### 6.2 回顾与沉淀

- 在 `cases/feature-xxx/summary.md` 中记录：
  - 实际实现与原计划的差异。
  - 决策点与经验教训。
  - 对 Agent 协作方式的改进建议（例如：哪个 Prompt 需要调整）。
- 这些文档会成为未来团队成员的「最佳实践样本」。

---

### 7. OpenClaw Studio 中的典型操作视图

- **需求输入区**：
  - 用于粘贴/编辑需求描述。
  - 选择关联的仓库/分支。
- **规划视图**：
  - 展示 `plan.md`/`plan.json`。
  - 支持勾选/调整子任务顺序与优先级。
- **执行视图**：
  - 针对子任务，显示 CodingAgent 生成的补丁列表。
  - 支持「应用补丁」「重新生成」「部分接受」等操作。
- **测试/评审视图**：
  - 显示 TestAgent 输出的问题、测试建议与验收清单。
  - 可勾选已完成项，记录实际结果。
- **历史与知识库视图**：
  - 按需求/Case 查看完整闭环（需求 → 计划 → Patch → 测试 → 上线 → 总结）。

---

### 8. 第一阶段你可以实践的最小闭环

建议你先用一个简单但完整的需求，走完下面流程：

1. 选一个小功能或重构需求（例如「为现有 API 增加一个可选参数」）。
2. 按本流程写需求文档 → 用 PlanningAgent 生成计划。
3. 拆成 2–3 个子任务，让 CodingAgent 生成补丁并人工应用。
4. 用 TestAgent 生成测试建议，实际跑一遍测试与手工验收。
5. 完成上线，并在 `cases/` 下形成完整案例。

完成 3–5 个这样的闭环后，你就有了一套更成熟、可以教给未来团队成员的工作流范式。
