# 规划视图功能说明

## "生成计划"按钮的应用场景

### 1. 主要用途
"生成计划"按钮用于**从需求描述自动生成实现计划**。这是 AI 原生开发流程的第一步。

### 2. 使用时机
- ✅ **新建案例后**：创建了一个新的需求案例，需要生成实现计划
- ✅ **需求变更后**：需求描述被更新，需要重新生成计划
- ✅ **计划缺失时**：当前案例还没有计划，需要生成

### 3. 工作流程

```
点击"生成计划"按钮
    ↓
前端发送请求到后端 API: POST /api/v1/cases/{caseId}/planning
    ↓
后端调用 PlanningAgent.generate_plan()
    ↓
PlanningAgent 执行以下步骤：
  1. 收集项目结构信息
  2. 读取相关文件内容（如果有）
  3. 构建系统提示词和用户消息
  4. 调用 LLM（使用 planning 模型）
  5. 解析 LLM 响应，提取：
     - plan_markdown: Markdown 格式的实现计划
     - tasks: 结构化任务列表（JSON 格式）
    ↓
后端保存计划到数据库和文件系统
    ↓
前端刷新显示计划内容和任务列表
```

### 4. PlanningAgent 的输入
- `requirement_description`: 需求描述（来自案例的 description 字段）
- `related_files`: 相关文件列表（可选）
- `codebase_overview`: 代码库概览（自动收集）
- `constraints`: 约束条件（可选）
- `acceptance_criteria`: 验收标准（可选）

### 5. PlanningAgent 的输出
- `plan_markdown`: Markdown 格式的实现计划，包含：
  - 需求分析
  - 技术方案
  - 实现步骤
  - 风险评估
  - 测试建议
- `tasks`: 结构化任务列表，每个任务包含：
  - `id`: 任务唯一标识
  - `title`: 任务标题
  - `description`: 任务描述
  - `related_files`: 关联文件列表
  - `risk_level`: 风险等级（low/medium/high）

## "任务列表"何时有数据？

### 1. 数据来源
任务列表的数据来自 **PlanningAgent 生成计划时提取的结构化任务信息**。

### 2. 任务提取机制

PlanningAgent 会尝试从 LLM 响应中提取任务列表：

```python
# PlanningAgent._extract_tasks_from_response()
# 1. 查找 JSON 代码块
json_match = re.search(r"```json\s*(\[.*?\])\s*```", content, re.DOTALL)
# 2. 解析 JSON
tasks_data = json.loads(json_match.group(1))
# 3. 转换为 Task 对象
```

### 3. 任务列表有数据的条件

✅ **成功情况**：
- LLM 返回的 Markdown 计划中包含 JSON 格式的任务列表
- JSON 格式正确，可以被成功解析
- 任务数据包含必需的字段（id, title, description 等）

❌ **空列表情况**：
- LLM 响应中没有 JSON 代码块
- JSON 格式错误，无法解析
- LLM 只返回了 Markdown 计划，没有结构化任务数据

### 4. 如何确保任务列表有数据？

#### 方法 1：优化 LLM 提示词（推荐）
在 PlanningAgent 的系统提示词中明确要求 LLM 返回 JSON 格式的任务列表：

```python
system_prompt = """...
请确保：
- 计划具体、可执行
- 子任务粒度适中（每个任务 1-3 小时工作量）
- 明确标注风险等级和依赖关系
- 提供测试建议和验收标准
- **必须在计划末尾包含 JSON 格式的任务列表**，格式如下：
```json
[
  {
    "id": "task-001",
    "title": "任务标题",
    "description": "任务描述",
    "related_files": ["file1.py", "file2.py"],
    "risk_level": "medium"
  }
]
```
"""
```

#### 方法 2：手动补充任务
如果任务列表为空，用户可以：
1. 查看 Markdown 计划中的任务描述
2. 在"计划"标签页中编辑计划，添加 JSON 格式的任务列表
3. 保存后，系统会尝试重新解析任务

#### 方法 3：使用任务管理 API
如果任务列表为空，可以通过 API 手动创建任务：
- `POST /api/v1/cases/{caseId}/tasks` - 创建任务
- `PUT /api/v1/cases/{caseId}/tasks/{taskId}` - 更新任务

### 5. 任务列表的显示

任务列表显示在"任务列表"标签页中，包含：
- **状态**：待处理（pending）/ 已完成（completed）
- **标题**：任务标题
- **风险等级**：低/中/高（带颜色标签）
- **关联文件**：任务相关的文件列表
- **操作**：标记完成、查看详情

### 6. 任务状态管理

- **初始状态**：所有任务默认为 `pending`（待处理）
- **更新状态**：用户可以点击"标记完成"按钮将任务状态改为 `completed`
- **状态同步**：任务状态保存在数据库中，刷新页面后仍然保持

## 完整工作流程示例

### 场景：实现一个新功能

1. **创建案例**
   - 在需求中心点击"创建案例"
   - 填写标题和需求描述
   - 保存

2. **生成计划**
   - 进入规划视图
   - 点击"生成计划"按钮
   - 等待 PlanningAgent 生成计划（可能需要几秒到几十秒）

3. **查看计划**
   - 在"计划"标签页查看 Markdown 格式的实现计划
   - 可以编辑和保存计划

4. **查看任务列表**
   - 切换到"任务列表"标签页
   - 如果任务列表有数据，可以看到所有子任务
   - 如果为空，需要手动补充或重新生成计划

5. **执行任务**
   - 在"执行视图"中选择任务
   - 生成代码补丁
   - 应用补丁

6. **测试和验收**
   - 在"测试视图"查看测试建议
   - 完成验收清单

7. **查看历史**
   - 在"历史视图"查看完整的开发历史

## 常见问题

### Q: 为什么任务列表是空的？
A: 可能是因为 LLM 响应中没有包含 JSON 格式的任务列表。可以尝试：
- 重新生成计划
- 手动编辑计划，添加 JSON 任务列表
- 检查 LLM 配置和 API 连接

### Q: 可以手动添加任务吗？
A: 可以。目前可以通过编辑计划 Markdown 添加 JSON 任务列表，或者使用 API 直接创建任务。

### Q: 任务列表和计划 Markdown 的关系？
A: 
- 计划 Markdown 是完整的实现计划文档
- 任务列表是从计划中提取的结构化数据
- 两者可以独立编辑，但建议保持同步

### Q: 如何更新任务状态？
A: 在"任务列表"标签页中，点击任务行的"标记完成"按钮即可更新状态。
